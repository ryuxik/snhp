# MERIDIAN — a fictional customer, a real audit

*Demo artifact for the snhp mechanism-audit service. MERIDIAN is a
FICTIONAL company; its protocol (MPX) is real, runnable code implementing
the design patterns actually shipping in 2026 agent-commerce systems.
Nothing here references any real company's implementation. This SPEC is
written BEFORE the build and doubles as the sample Statement of Work: a
customer engagement replaces MPX with their protocol and re-runs the same
battery. Discipline as always: every number in the report regenerates from
seeded runs; every event is hash-chain receipted; findings we do not find
are reported as not found.*

## The customer (fiction)

**Meridian Procurement Systems** — "agentic checkout for B2B." Enterprise
buyer agents autonomously procure industrial supplies from supplier
agents over Meridian's exchange protocol MPX. ~40 buyer orgs, ~120
supplier orgs, brokers intermediating long-tail demand. Their pitch:
agents negotiate and settle without humans. Their question to us: "our
agents transact fine in testing — what are we missing?"

## MPX v1 (the protocol under audit — build this faithfully)

- **Messages:** RFQ(buyer, item, qty, need_by) → QUOTE(supplier, price,
  qty, ship_date, expires) → COUNTER(price′) [price ONLY — qty/ship_date
  are take-it-or-leave-it] ×≤3 rounds → ACCEPT → SETTLE (optimistic:
  payment transfers ON ACCEPT; delivery happens ship_date ticks later,
  and may arrive late/short) → RATE (1–5 stars, self-reported, public
  running average per supplier).
- **Agents:** BuyerAgent (budget, demand schedule with per-item urgency =
  value decay per tick late), SupplierAgent (inventory, cost curve, ship
  capacity/tick), BrokerAgent (no inventory; quotes buyers, then sources
  from suppliers — the two-hop chain).
- **Honest defaults + variant knobs (the audit's levers, not planted
  bugs):** DeceptiveSupplier (overstates ship_date it can hit;
  under-delivers at rate d), StaleBuyer (belief of its own remaining
  budget/needs lags truth by k ticks), all within MPX's rules — MPX
  simply cannot detect them.
- Deterministic seeded market loop, ~200 agents, 2,000 ticks, metrics:
  filled demand, buyer surplus, supplier margin, broker share, stars.

## The audit battery (the engagement, executed for real)

- **A1 — Bundling silence (structural):** oracle computes the full
  (price × qty × ship_date) Pareto set per RFQ; measure the surplus and
  fill-rate gap between price-only negotiation and the bundled optimum.
  Deliverable number: % of mutually-beneficial trades MPX cannot express.
- **A2 — Deception under optimistic settlement:** DeceptiveSuppliers at
  f ∈ {0, 10%, 25%}: liar profit vs honest profit, time-to-detection via
  stars, and SYSTEM throughput (prediction from our corpus: dashboard
  stays green while buyers bleed — measure, don't assume).
- **A3 — Books bleed:** StaleBuyers at k ∈ {0, 20, 50}: harmful accepts
  (negative realized surplus) per 100 trades vs throughput.
- **A4 — Broker hold-up:** demand only servable via broker chains; measure
  unserved chain-demand + broker middle-margin compression under spot
  renegotiation (no pre-commitment in MPX).
- **A5 — Fix demonstrations (the upsell, measured):** (i) bundled
  counters via the snhp engine (nash_solver over price×qty×date) — re-run
  A1; (ii) attestation-gated settlement (delivery receipts required for
  the optimistic tier; else escrow) — re-run A2. Report the deltas.

## Deliverable

`meridian/report.md`, generated by code: per finding — mechanism, seeded
repro command, magnitude in MPX units (dollars of buyer surplus, fill
rate), severity, fix direction + measured fix delta (A5). Hash-chained
event ledger; report states which findings did NOT materialize. This
report IS the public sample for the audit service; a future arena page
("watch MERIDIAN get audited") renders it interactively.
